## 1. 本書の目的

本書は、I2I工程支援Webアプリの **Webデザイン方針・UI原則・画面レイアウト・視覚的一貫性ルール** を定義するデザイン基準書である。  
本書は補助資料ではなく、**実装工程・レビュー工程・保守工程・別セッション引き継ぎ工程における正式成果物** として扱う。

本書の目的は以下とする。

1. イラスト制作工程に沿った GUI を一貫した形で実現する
2. レスポンシブな操作体験を維持する
3. Microsoft Fluent 2 のデザイン原則を、本プロジェクト向けに実装可能なルールへ落とし込む
4. `SKILL.md` と責務分離し、見た目・レイアウト・状態表現の判断基準を集約する
5. 別セッションでも、同じ UI/UX 方針で作業継続できる状態をつくる

---

## 2. デザインの上位方針

本アプリのWebデザインは **Microsoft Fluent 2** の design principles、layout、typography、motion、design tokens の考え方を基礎とし、本プロジェクト固有の **イラスト制作工程GUI** に合わせて再構成する。

ただし、本アプリの目的は Microsoft 製品の外観を模倣することではない。  
Fluent 2 を **原則・構造・トークン設計の基準** として採用し、本アプリ固有の業務導線・制作フロー・I2I操作体験を優先してデザイン判断を行う。

## 3. Fluent 2 採用方針

### 3.1 採用対象
本アプリでは Fluent 2 の以下を採用対象とする。

- Design principles
- Layout
- Typography
- Motion
- Design tokens
- コンポーネント設計思想
- レスポンシブ設計の考え方
- アクセシビリティへの配慮

### 3.2 非採用 / そのまま模倣しない対象
以下はそのままコピーしない。

- Microsoft 製品と同一の外観再現
- Fluent 2 コンポーネントの全面的な固定利用
- ブランド専用の表現を無批判に転用すること

### 3.3 本アプリ向け再解釈
Fluent 2 の原則を、本アプリでは以下のように再解釈する。

| Fluent 2 の考え方 | 本アプリでの解釈 |
|---|---|
| Natural on every platform | PC / タブレット / モバイルで自然に使える UI にする |
| Built for focus | 今の工程・今必要な入力・次の操作を明確に見せる |
| One for all, all for one | 幅広い利用者が迷いにくい、読みやすい、操作しやすい UI にする |
| Unmistakably Microsoft | Microsoft らしさの模倣ではなく、一貫したデザインルールで信頼感を出す |

## 6. 情報設計の基本方針

### 6.1 画面の主軸
画面は、以下の情報順序を基本とする。

1. 現在の工程
2. 工程説明
3. 必須入力
4. 補助選択（タグ、テンプレート差し込み）
5. 生成されるプロンプト
6. 実行操作
7. 結果画像
8. 比較・再実行・次工程への遷移

### 6.2 認知負荷の制御
1画面で扱う主目的は 1 つに寄せる。  
ただし、前後工程とのつながりは常時見える状態を維持する。

### 6.3 見せる順番
「何を見るか」より「何をすればよいか」が分かる構成を優先する。

## 7. 画面構造の原則

### 7.1 全体フレーム
全画面の基本構成は以下とする。

- ヘッダー
- 工程ナビゲーション
- メインコンテンツ
- 補助パネル（必要時）
- フッターまたは補助操作エリア

### 7.2 メインコンテンツの構成
工程画面の基本構成は以下とする。

1. 工程ヘッダー
2. 入力画像エリア
3. 工程別設定エリア
4. タグ / 単語リスト選択エリア
5. プロンプト生成エリア
6. I2I 実行エリア
7. 結果表示エリア
8. 次アクションエリア

### 7.3 レイアウト密度
- 過密を避ける
- 1セクションの責務を明確にする
- 強調と静けさのバランスを取る
- 背景装飾より情報の関係性を優先する

## 8. イラスト制作工程フローに沿ったGUI原則

### 8.1 工程定義
本アプリでは少なくとも以下の工程を扱う。

- No.4 構図ラフ
- No.5 大ラフ / ポーズラフ
- No.6 線画寄り / クリーンアップ
- No.7 カラーラフ
- No.8 塗り込み / 描き込み
- No.9 仕上げ

工程の意味やプロンプトルールは `SKILL.md` に定義し、本書では **見せ方・導線・UI表現** を定義する。

### 8.2 ステップ表示
- 画面上部に工程ステッパーを常時表示する
- 現在の工程を強調表示する
- 完了済み工程は再訪可能とする
- 未完了工程へのジャンプは可能だが、注意表示を出せる設計とする
- 工程ごとの入力差分は画面構造を大きく変えずに吸収する

### 8.3 各工程画面の順序固定
各工程では、基本的に以下の順序を固定する。

1. 工程の説明
2. 入力画像の確認
3. タグやパラメータの選択
4. 生成プロンプトの確認・編集
5. 実行
6. 結果比較
7. 保存 / 次工程へ進行

### 8.4 ユーザーの現在地
常に以下が分かる状態にする。

- 今どの工程にいるか
- 前工程は何か
- 次工程は何か
- 今の操作で何が変わるか

## 9. レスポンシブデザイン方針

### 9.1 前提
本アプリはレスポンシブデザインを採用する。  
単なる縮小ではなく、利用端末ごとに **情報優先順位** を保った再配置を行う。

### 9.2 デバイス別方針
#### Desktop
- 主利用環境
- 2〜3 カラムを許容
- 画像比較とプロンプト確認を同時表示しやすい構成にする

#### Tablet
- 2 カラム中心
- 比較表示は上下配置または切替表示も許容
- タップ操作のしやすさを優先する

#### Mobile
- 1 カラム前提
- 工程ナビは圧縮表示または横スクロール表示を許容
- 長いフォームはアコーディオンまたはセクション折りたたみで整理する
- 実行操作・保存操作は常に押しやすい位置に置く

### 9.3 タッチ領域
- モバイルでの最小タップ領域を十分確保する
- 誤操作しやすい密集配置を避ける

## 10. レイアウトルール

### 10.1 スペーシング基準
Fluent 2 の spacing ramp の考え方に合わせ、4px ベースの余白設計を採用する。  
本プロジェクトでは以下を主要値として使う。

- 4px
- 8px
- 12px
- 16px
- 20px
- 24px
- 32px
- 40px
- 48px

### 10.2 推奨利用
- ラベルと入力欄: 8px
- フィールド間: 12〜16px
- カード内余白: 16px
- セクション内余白: 16〜24px
- セクション間: 24〜32px
- 主要レイアウト余白: 24px 以上

### 10.3 グリッド
- Desktop は 12 カラム系の考え方を採用してよい
- Tablet は 8 カラム相当の運用を許容
- Mobile は 4 カラム相当または 1 カラムに最適化する
- グリッドは固定ではなく、コンテンツ理解を優先して調整する

### 10.4 視線誘導
余白と情報のまとまりで階層を作る。  
罫線や強い背景色に頼りすぎない。

## 11. タイポグラフィ方針

### 11.1 基本方針
タイポグラフィは、装飾ではなく **情報階層の明示** を目的とする。  
同じ役割の文字は、全画面で同じ見え方にする。

### 11.2 文字階層
最低限、以下の階層を定義する。

1. ページタイトル
2. セクションタイトル
3. カードタイトル
4. フィールドラベル
5. 本文
6. 補助説明
7. エラー / 注意
8. キャプション

### 11.3 ルール
- 強調はサイズ・太さ・余白で表現する
- 色だけで階層を作らない
- 長文説明は読みやすい行間を取る
- ラベルは短く明確にする
- 英単語タグと日本語UIの混在を考慮する

---

## 12. カラー方針

### 12.1 基本方針
配色は、作品そのものではなく **UI の理解促進** を目的とする。  
画像表示領域と競合しない、落ち着いた中立ベースを採用する。

### 12.2 原則
- 背景は中立色を基調とする
- 主操作はアクセントカラーで明確にする
- 状態色は意味を固定する
- 結果画像の色を邪魔しない UI 配色にする

### 12.3 状態色
最低限、以下を定義する。

- Primary
- Secondary
- Success
- Warning
- Danger / Error
- Info
- Disabled
- Focus

### 12.4 工程色
工程ごとの色分けは補助的に使ってよいが、色だけで工程識別しない。  
アイコン・ラベル・位置でも識別できるようにする。

### 13.1 主要コンポーネント
本アプリで最低限必要な主要 UI コンポーネントは以下とする。

- Header
- Stepper / Progress indicator
- Stage selector
- Card
- Panel
- Form field

- Toggle
- Button
- Icon button
- Modal / Dialog
- Toast / Notification
- Loader / Progress
- Image preview

- Tabs / Accordion（必要時）

### 13.2 コンポーネント共通ルール

- 状態変化を持つコンポーネントは、通常 / ホバー / アクティブ / 選択 / 無効 / エラーを設計する
- 装飾より可読性を優先する

## 14. タグ選択UI方針

### 14.1 役割
タグUIは、本アプリにおける重要入力装置である。  
単なる装飾ではなく、**工程別プロンプト生成のための構造化入力** として扱う。

### 14.2 表現方針
- カテゴリごとに整理する
- 選択状態が明確に分かるようにする
- 検索と一覧の両方を用意できる設計にする
- 選択したタグがプロンプトへどう反映されるか分かるようにする

### 14.3 禁止事項
- 1画面に無秩序に全タグを並べる
- 選択状態が色だけでしか分からない
- 何に使われるタグか不明な状態にする

---

## 15. フォーム設計方針

### 15.1 原則
- フィールド名は明確にする
- 必須 / 任意を明示する
- 説明文は短く、必要時のみ展開可能にする
- 入力内容が結果へどう影響するか分かるようにする

### 15.2 APIキー入力
追加要望に基づき、API キーは **ユーザー自身が用意し、Web サイト側では保管しない**。  
UI 上では以下を満たす。

- API キー入力欄を設定画面または初期設定画面に用意する
- マスク表示 / 表示切替を用意する
- ローカルキャッシュ保存の有無をユーザーが制御できるようにする
- サーバ保存しないことを明記する
- 消去操作を明示的に用意する

### 15.3 プロンプト確認欄
- 自動生成結果を編集可能にする
- テンプレ適用部分と手入力部分の関係が分かるようにする
- コピーボタンやリセットを用意できる構造にする

---

## 16. 画像表示・比較表示方針

### 16.1 入力画像表示
- 工程の元画像であることが分かる表示にする
- 画像の縮小時でも、元比率を極力維持する
- 必要に応じて拡大確認できる構造にする

### 16.2 結果画像表示
- 最新結果が分かるようにする
- 履歴比較の拡張余地を持たせる
- 実行中 / 失敗 / 成功の状態を明確に分ける

### 16.3 比較表示
- 入力と出力の並列比較を基本とする
- モバイルでは切替比較も許容する
- 「何を元に何が出たか」が分からなくならないようにする

## 17. 状態表現ルール

### 17.1 必須状態
すべての主要 UI で、必要に応じて以下の状態を設計する。

- Default
- Hover
- Focus
- Active
- Selected
- Disabled
- Loading
- Success
- Warning
- Error

### 17.2 原則
- 状態は色だけでなく、形・ラベル・アイコンでも補助する
- ローディング中は操作不能範囲を明確にする
- エラーは原因と対処をテキストで示す

---

## 18. モーション方針

### 18.1 基本方針
モーションは演出のためではなく、**状態変化・画面遷移・関係性の理解補助** のために使う。

### 18.2 許容されるモーション
- 工程遷移時の軽いフェード / スライド
- モーダルの自然な出入り
- タグ選択時の軽い応答
- ローディングの穏やかな視覚化
- 比較表示切替時の控えめな遷移

### 18.3 禁止寄り事項
- 常時動き続ける装飾
- 強すぎるアニメーション
- 作業集中を妨げるモーション
- 目的不明の派手演出

---

## 19. アクセシビリティ方針

### 19.1 原則
- キーボード操作可能
- フォーカス可視化
- 色だけに依存しない状態表現
- 十分な可読性
- モバイルでも押しやすい操作領域
- エラーメッセージは意味が分かる文章にする

### 19.2 優先事項
- 入力・実行・保存が迷わず行える
- ステップ進行が読み上げや視認で理解できる
- タグ選択や比較表示が過度に複雑にならない

---

## 20. デザイントークン方針

### 20.1 原則
色、余白、文字、角丸、影、モーションなどはハードコードを避け、デザイントークンで管理する。

### 20.2 想定トークンカテゴリ
- Color
- Space
- Typography
- Radius
- Border
- Shadow / Elevation
- Motion
- Z-index
- Breakpoint

### 20.3 命名例
- `color.background.page`
- `color.background.surface`
- `color.background.card`
- `color.text.primary`
- `color.text.secondary`
- `color.border.default`
- `color.status.success`
- `space.100`
- `space.200`
- `radius.medium`
- `shadow.card`
- `motion.duration.fast`

### 20.4 実装ルール
- 同じ意味の値は同じトークンを使う
- 一時しのぎの直接指定を増やさない
- テーマ切替や将来拡張を見越した命名にする

## 21. Microsoft系Webアプリデザインパターンとの整合方針

### 21.1 方針
本アプリは Microsoft の Web アプリ設計思想と整合するように、以下を重視する。

- 明確な情報階層
- 操作の一貫性
- レスポンシブ性
- アクセシブルな導線
- 状態が分かる UI
- 役割ごとの責務分離

### 21.2 実装上の解釈
- 画面全体はアプリケーションとして整理されたフレームを持つ
- 設定、実行、結果、履歴の責務を分ける
- 工程別に目的が変わっても、画面骨格は維持する

## 22. ダークモード / テーマ拡張方針

### 22.1 初期方針
初期版ではライトテーマ優先でもよいが、将来的なテーマ切替を前提に設計する。

### 22.2 ルール
- 直値で色を埋め込まない
- 画像表示に悪影響の少ないテーマ設計を考慮する
- 結果画像と UI の境界が見失われないようにする
