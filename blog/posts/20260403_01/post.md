# 軍儀のブラウザゲームを作って学んだこと

『HUNTER×HUNTER』に登場する軍儀を題材に、Codexでブラウザゲームを作った。最初は「重ね置きもある独特なゲームを、ブラウザで気持ちよく遊べる形にしたい」という発想から始まったが、進めてみると学びが多かったのはゲーム性そのものより、設計と改善の進め方だった。

今回は、実際に確認した3つの流れに沿って振り返る。

## 0. ChatGPT の「軍儀ゲームWeb実装設計」

最初の設計段階で効いたのは、「軍儀を作る」という曖昧な目標を、そのまま実装可能な要件に変換したことだった。

- GitHub Pages で公開できる静的サイトにする
- `React + TypeScript + Vite + Three.js + react-three-fiber` を使う
- CPU 戦のみ
- 9x9 盤
- 重ね置きを 3D で表現する
- `applyMove()` に状態変更を集約する
- ルールエンジンと UI を分離する
- 駒の移動定義はデータ駆動にする
- まずは入門編互換、あとから上級編互換へ拡張できる構造にする
- CPU は将来的に Web Worker 化しやすい設計にする

さらに、実装順まで先に決めていたのが大きかった。ゲーム状態、合法手生成、`move/stack/drop/capture`、3D 盤面、入力 UI、CPU、ログ、テスト、上級編対応という順番が見えていたので、途中で「次に何をやるべきか」がぶれにくかった。

題材が特殊なボードゲームほど、最初に要件を文章で固定しておく価値は大きいと感じた。

## 1. 作成Webブラウザ軍儀ゲームの基盤

最初の実装セッションでは、見た目を急ぐより先に、壊れにくい土台を作る方針が取られていた。ここが今回いちばん効いた部分だったと思う。

早い段階で分離されていたのは、主に次の要素だった。

- 合法手生成や勝敗判定を持つルールエンジン
- 駒定義
- 入門編と上級編を切り替える `ruleset`
- LocalStorage 保存
- CPU 思考処理
- 3D 盤面 UI

軍儀は、ただ駒を動かすだけではなく、重ねる、持ち駒から置く、詰みを判定する、投了を扱う、といった処理が多い。だからこそ、UI から直接状態を書き換えず、`applyMove()` に一本化した設計はかなり強かった。

また、CPU を最初から Web Worker 化しやすい境界で切っておいたのも良かった。ブラウザゲームでは AI の強さだけでなく、思考中に画面を固めないことも体験の一部だからだ。

一方で、最初の版ではあえて暫定対応も入っていた。たとえば入門編の初期配置は、手駒からの `新` を検証しやすくするために、完全再現より試しやすさを優先していた。ここから学んだのは、最初から完璧を目指すより、まず遊べる状態まで持っていく方が結果的に前に進みやすいということだった。

## 2. 提示GUNGI-WEB改善案

基盤ができたあとに改善案を整理すると、出てきたのは AI の強化だけではなかった。

- リプレイと解析モード
- 駒の動きが分かるルール表示
- 初回ガイドやおすすめ配置
- 自動対局での先手後手別 CPU 難易度
- 対局時間や各手の所要時間
- 保存管理と局面エクスポート
- 自動対局ログの外部保存
- 盤面 UI の操作性改善
- E2E テスト
- ドキュメント整備

実際に今のゲームを見ると、対局状況への手数と経過時間の表示、リプレイ、保存管理、解析ヒント、自動対局、ログ保存の土台など、改善案がかなり具体的な機能として反映されている。

ここで強く感じたのは、この種のゲームは「正しく動く」だけでは足りず、「理解しやすい」「振り返れる」「比較できる」機能が入って初めて完成度が上がるということだった。軍儀のようにルール把握のハードルがあるゲームでは、強い CPU そのものより、学習導線やログの方が価値を持つ場面も多い。

## 作ってみて残った学び

1. 曖昧なアイデアを、実装できる要件に変換する作業が最重要だった。
2. ボードゲームは UI 先行ではなく、ルールエンジン先行で作った方が後から伸ばしやすい。
3. `ruleset` 分離とデータ駆動の駒定義は、入門編から上級編へ広げるための保険になる。
4. ブラウザ AI は、強さと同じくらい非同期で快適に動くことが重要だった。
5. GitHub Pages 前提の制約は、相対パスや静的ビルドだけでなく、サーバ機能をどう外出しするかまで設計に影響した。
6. 改善案を Issue として言語化すると、思いつきがそのまま次の実装計画になる。

## まとめ

今回の軍儀ブラウザゲーム制作で面白かったのは、設計、試作、改善の3段階がそのまま学習プロセスになっていたことだった。

特殊ルールのゲームを Web で作るときは、最初から全部を完成させようとするより、まずはルールと状態管理の土台を正しく切り、そのあと実際に遊びながら改善点を拾っていく方がうまく進む。Codex と一緒に進める開発でも、この進め方はかなり相性が良かった。
